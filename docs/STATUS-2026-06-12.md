## 一、项目定位（已对齐）

### 1.1 三层用户金字塔

```
华义匀安内部（最先用）
   ↓ 验证稳定后
IP-Guard 客户（外部，受控开放）
   ↓ 产品成熟后
独立产品卖给其他公司（多租户 SaaS / 私有化）
```

### 1.2 4 个核心场景（按优先级）

| 优先级 | 场景 | 用户 | 类型 |
|--------|------|------|------|
| P0 | 实施/远程售后查产品配置策略 | 实施 + 售后 | RAG 问答 |
| P0 | 根据模块 + 模板生成介绍 PPT | 售前 | RAG + 文档生成 |
| P0 | 拉取相似客户对比 PDF | 销售 | RAG + 文档生成 |
| P1 | 客户自主查策略配置 | 外部客户 | RAG 问答（脱敏）|

### 1.3 4 个真实痛点（驱动需求）

1. 售后每次都翻一遍同样的 PDF
2. 新员工上手 IP-Guard 产品慢
3. 不同客户需要不同的 PPT 介绍，人工制作浪费时间
4. 销售面见客户时不能快速做对比图

### 2.2 十条迁移友好铁律（CLAUDE.md v1.2）

1. 所有模型调用走 `app.services.llm` 统一抽象层
2. 配置 100% 走环境变量
3. 使用 OpenAI-Compatible 协议
4. Embedding 用 bge-m3（本地）
5. Prompt 写到 `prompts/` 目录
6. 关键参数走 `config.yaml`
7. 保留 Ollama 兼容性（每周五验证）
8. Agent 编排走 LangGraph，RAG 工具走 LlamaIndex
9. CRM 调用走统一抽象层（`services/crm.py`）
10. Agent 工具集物理隔离（EXTERNAL_TOOLS / INTERNAL_TOOLS + 4 层防御）

- P1：API 响应脱敏

## 五、本项目是不是多智能体 RAG？

### 5.1 多智能体 RAG 5 种形态

| 形态 | 描述 | 代表 |
|------|------|------|
| 1. 单 Agent 工作流 | LLM + 固定状态机 | naive RAG / RAGFlow |
| 2. 单 Agent + 工具调用 | LLM 自主选工具 | ChatGPT plugins |
| 3. Router + 专用 Agent | Router 判意图 → 分发 | LangGraph conditional_edges |
| 4. Supervisor 多 Agent | 中央协调多 Worker | AutoGen / CrewAI |
| 5. Swarm 协作 | Agent 直接通信无中央 | LangGraph Swarm |

**"多智能体"门槛在形态 3**。

### 5.2 当前位置 + 演进路径

| 阶段 | 形态 | 是不是多智能体 RAG |
|------|------|------------|
| W2-W4 lean MVP（ConfigQA 单 Agent）| 形态 1 | ❌（agentic RAG） |
| Phase 2-4 完整版（Router + 4 个角色专用 Agent）| 形态 3 | ✅ |
| Phase 5+ 终态（Supervisor 协调）| 形态 4 | ✅ |

### 5.3 架构上已经预埋的多智能体基础

| 基础设施 | 状态 |
|---------|------|
| LangGraph（铁律 #8）| ✅ |
| `agents/factory.py` build_agent(user) | ✅ |
| `tool_registry.py` EXTERNAL/INTERNAL TOOLS | ✅ |
| 铁律 #10 工具集物理隔离 | ✅ |
| `services/llm.py` 抽象层 | ✅ |
| #71 工具集断言测试 | ✅ |

### 5.4 对外话术

> 当前 MVP 是 agentic RAG（单 Agent 工作流 + 引用 + 拒答），架构层已为多智能体预埋 Router + 工具集物理隔离 + Agent 工厂。Phase 2-4 落地 Router + 4 个角色专用 Agent 后升级为完整的多智能体 RAG。**企业治理（10 条迁移友好铁律 / 4 IdP / 客户级 ACL / 多租户预留）是我们跟 AutoGen / CrewAI 等开源方案的核心区分点。**

## 六、关键决策记录

### 6.1 不切 Dify / RAGFlow 全栈

- 表面省 4-6 周，实际净亏 3-5 周（客户级 ACL + 4 IdP + 多租户改造比自建慢）
- license 隐患：Dify 商业 SaaS 转售需付费
- 铁律 #1 / #7 失效（LLM 切换从改 .env 变成进 Dify 界面）

### 6.2 用 OpenWebUI 套壳做 lean MVP UI

- 节省 2-3 周自写 Next.js
- 通过 OpenAI 兼容协议解耦，不 fork 源码
- 升级风险可控（pinned 镜像 tag）

### 6.3 多租户改造推迟

- 当前所有表不加 `tenant_id`
- Phase 5+ 产品化决策时再改造
- 单租户 MVP 先跑通

### 6.4 接受 PR #14/#15/#16/#17 的 NEEDS_REVIEW 自合并

- 用户人工授权，非流程违规
- Codex 自审质量 9-10/10，Part D 触发后用户已读 Handoff
- 每个 PR 补 post-merge review 归档（PR-10/14/15/16）

### 6.5 #25 spec 分裂为两个版本

- v1（客户档案抽取）→ 移到 `phase2/W5-25b-customer-doc-extract.md`，Phase 2+ 启用
- v2（产品 KB 抽取）→ `W2-D2-25-product-kb-metadata-extract.md`，lean MVP 立刻做
