## 1. 必须修复（Blocker）

### B1. RustFS 环境变量名错误导致 Web Console 登录失败

- **位置**：`docker-compose.yml:48-50`
- **问题**：当前用 `RUSTFS_ACCESS_KEY` / `RUSTFS_SECRET_KEY`，套了 MinIO 命名风格。RustFS 官方实际识别的是 `RUSTFS_ROOT_USER` / `RUSTFS_ROOT_PASSWORD`。结果是容器忽略你设置的密码，回退到内置默认 `rustfsadmin`/`rustfsadmin`。用户已**实测无法用 `rustfsadmin-please-change-me` 登 9001 控制台**。Integration tests 之所以通过，是因为 boto3 客户端用了同样的错误密码——双方都"错对了"，掩盖了真问题。这就是绿色测试掩盖真错误的典型场景。
- **期望修法**：
  ```yaml
  rustfs:
    environment:
      RUSTFS_ROOT_USER: rustfsadmin
      RUSTFS_ROOT_PASSWORD: rustfsadmin-please-change-me
      RUSTFS_VOLUMES: /data
  ```
  修完必须执行 `docker compose down rustfs && docker compose up -d rustfs`（recreate 而非 restart），然后**手动验证 http://localhost:9001 用 rustfsadmin / rustfsadmin-please-change-me 能登入**，截图或粘贴验证日志到 fix_attempt。
- **依据**：RustFS 官方 README + deploy docs；新增反模式 G1（见下文 §6）。

### B2. `get_storage()` 无生命周期管理，每次调用重建 client 并发同步 IO

- **位置**：`backend/app/services/storage.py:60-70, 166-168`
- **问题**：`get_storage()` 每次返回 `S3CompatibleStorage()` 新实例，`__init__` 里同步执行 `boto3.client(...)` + `_ensure_bucket()`（含一次 `head_bucket` HTTP 往返）。一旦 FastAPI DI 用 `Depends(get_storage)`，每个请求都会重建 client 并阻塞 event loop 完成一次网络 IO。Codex 在 Handoff §5 / §E1 已自行识别，本轮升级为必修。注意：**不能用 `@lru_cache`**——命中 ANTIPATTERN E1。
- **期望修法**（推荐方案 a，FastAPI lifespan 单例）：
  ```python
  # backend/app/main.py
  from contextlib import asynccontextmanager
  from app.services.storage import S3CompatibleStorage

  @asynccontextmanager
  async def lifespan(app: FastAPI):
      storage = S3CompatibleStorage()
      await storage.ensure_ready()           # 显式异步 bucket 检查
      app.state.storage = storage
      yield

  app = FastAPI(lifespan=lifespan)

  # backend/app/services/storage.py
  class S3CompatibleStorage(DocumentStorage):
      def __init__(self) -> None:
          self._client = boto3.client(...)   # 不再调用 _ensure_bucket
          self._bucket = settings.storage_bucket

      async def ensure_ready(self) -> None:
          await to_thread.run_sync(self._ensure_bucket)

  def get_storage(request: Request) -> DocumentStorage:
      return cast(DocumentStorage, request.app.state.storage)
  ```
  - `__init__` 不再含 IO，`get_storage()` 不再触发网络调用。
  - 兼容现有测试：`S3CompatibleStorage()` 仍可直接构造，integration 测试在 fixture 里 `await ensure_ready()` 一次即可。
- **依据**：铁律 #1 抽象层的生命周期一致性 + ANTIPATTERN E1（避免 `lru_cache` 包 IO）。

### B3. `test_storage_error_on_invalid_endpoint` 用 monkeypatch 改 settings 单例 + 错误标 integration

- **位置**：`backend/tests/services/test_storage.py:1, 57-61`
- **问题**：
  1. `monkeypatch.setattr("app.services.storage.settings.storage_endpoint", "http://127.0.0.1:1")` 直接改 Pydantic v2 `BaseSettings` 单例字段。Pydantic v2 默认配置下要么抛 `ValidationError`，要么修改成功但 monkeypatch 还原不可靠 → **测试结束可能污染后续测试**。
  2. 文件顶部 `pytestmark = pytest.mark.integration` 一刀切标全文件。但 `test_storage_error_on_invalid_endpoint` **不需要真实 RustFS**（它就是测连接失败）。CI 跑 `-m "not integration"` 永远跳过它，实际等于无效测试。
- **期望修法**：
  ```python
  # 拆掉全文件 pytestmark，改为每个 integration 测试单独标
  # （或者把这个 unit 测试单独拆到 test_storage_unit.py）

  @pytest.mark.asyncio
  @pytest.mark.integration
  async def test_save_and_load_roundtrip() -> None: ...

  # 这个改成纯 unit，无需 integration mark
  def test_storage_error_on_invalid_endpoint(monkeypatch: pytest.MonkeyPatch) -> None:
      monkeypatch.setenv("STORAGE_ENDPOINT", "http://127.0.0.1:1")
      from app.core.config import Settings
      from app.services import storage as storage_module
      monkeypatch.setattr(storage_module, "settings", Settings())
      with pytest.raises(StorageError):
          S3CompatibleStorage()
  ```
- **依据**：ANTIPATTERN D1（测试共享 app/router 污染）的精神延伸；CI 必须真正跑该测试才有意义。

### B4. `_key` 不防路径穿越

- **位置**：`backend/app/services/storage.py:83-87`
- **问题**：`doc_id` 若来自用户上传文件名（Phase 2 #28 双轨入库会发生），传 `"../../secret"` 会被规约成 `documents/../../secret`，部分 S3 实现可能写到 bucket 外或覆盖其他对象。即使 RustFS 本身规约安全，**显式拒绝**才是正确防御。
- **期望修法**：
  ```python
  import re
  _DOC_ID_PATTERN: Final = re.compile(r"^[A-Za-z0-9._\-]+$")

  def _key(self, doc_id: str) -> str:
      normalized = doc_id.strip().lstrip("/")
      if not normalized:
          raise StorageError("Document id must not be empty")
      if not _DOC_ID_PATTERN.fullmatch(normalized):
          raise StorageError(f"Invalid doc_id (only [A-Za-z0-9._-] allowed): {doc_id}")
      return f"{_BUCKET_KEY_PREFIX}/{normalized}"
  ```
  加配套 unit 测试 `test_invalid_doc_id_rejected`（`../`、空串、含 `/`、含 `\\` 等 4 个 case）。
- **依据**：通用安全防御；Phase 2 #28 双轨入库会让 `doc_id` 从入口走到这里。

## G1：S3 兼容 ≠ 环境变量兼容

**场景**：切换 S3-compatible 对象存储服务（MinIO → RustFS / SeaweedFS / Garage / Ceph RGW 等）。

**反模式**：把 S3 API 兼容性误解为「管理面环境变量也兼容」，直接套用旧服务的变量命名风格。

**典型错误**：

1. boto3 客户端用同样错误密码访问 → integration tests **碰巧通过**（双方都"错对了"）

**正确做法**：
1. 查官方 README / Deploy doc 确认变量名（RustFS 是 `RUSTFS_ROOT_USER` / `RUSTFS_ROOT_PASSWORD`）
