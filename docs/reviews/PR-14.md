## 2. 建议改进（Nice to have，按优先排序）

### N1. LocalIdP `access_token == username`，Phase 2 ACL 中间件必须显式 reject

- **位置**：`backend/app/services/auth.py:106`
- **当前**：
  ```python
  return Token(access_token=username, expires_in=86400)
  ```
  `LocalIdP.exchange_code` 直接把 `username` 作为 `access_token` 返回；`get_user_info` 用同一个 username 字典查回 User。等于：**token 就是明文 username，没有签名 / 没有有效期校验 / 没有不可猜性**。
- **风险**：当前 `/api/v1/auth/login` endpoint 返回这个 token 给客户端。Phase 2 ACL 中间件如果 `Depends(get_current_user)` 简单复用 `LocalIdP.get_user_info(Token(access_token=raw_header))` 鉴权，等于**任何人猜中 username 字符串就能拿到该 user 的 ACL 上下文**——`is_external=False` / `role` / `allowed_depts` 全部白送。`LocalIdP` 受 production 防呆（auth.py:78）保护，开发期员工独享，可接受；**但合并前必须在 Handoff §7 显式告诉下一轮 ACL 实现者**。
- **建议修法**（本 PR 不修，加 §7 警告即可）：
  - Handoff §7 加一条"上下文 8：LocalIdP token 不安全 — Phase 2 ACL 中间件必须**拒绝接受 raw token 作为身份证明**。可选方案 a) JWT 签名 + 短 TTL；b) 服务端 session store + opaque token；c) 仅允许 LocalIdP 在 dev 环境跑 ACL bypass。**严禁直接拿 header 里的 token 字符串走 `get_user_info` 鉴权**。"
  - `auth.py:108-127` 的 `get_user_info` 顶部加 docstring 警告"This method is identity-mapping by design; production callers must validate signed tokens before invoking."
- **依据**：spec §11 / Handoff §7 是契约出口；当前 §7 上下文 1-7 没提 token 安全性，下一轮 ACL 实现者很容易踩坑。

### N2. `@lru_cache` 包 `get_settings()` 让 `monkeypatch.setattr` 测试间共享 Settings 单例

- **位置**：`backend/app/core/config.py:81-83`
- **当前**：
  ```python
  @lru_cache
  def get_settings() -> Settings:
      return Settings()

  settings = get_settings()
  ```
- **问题**：`@lru_cache` 让 `get_settings()` 全生命周期返回同一个 Settings 实例。当前 test_auth.py 用 `monkeypatch.setattr(auth_module, "settings", SimpleNamespace(...))` 绕开了这个问题（替换的是 `auth_module.settings` 引用，不是 `get_settings()` 缓存），所以**当前测试没踩雷**。
- **但是**：未来如果有任何代码用 `from app.core.config import get_settings; cfg = get_settings()`（典型场景：测试 fixture 想 fresh 一份 settings），缓存会让测试间状态泄漏。这不是 ANTIPATTERN E1（E1 针对的是 client / IO 对象，Settings 加 @lru_cache 是 pydantic-settings 官方推荐做法）。
- **建议**（可选）：
  - 加 conftest.py 全局 fixture：
    ```python
    @pytest.fixture(autouse=True)
    def _clear_settings_cache():
        get_settings.cache_clear()
        yield
        get_settings.cache_clear()
    ```
  - 或者在 ANTIPATTERNS.md 加一条新条目：「@lru_cache 包 Settings 工厂在测试间会泄漏 monkeypatch 状态，必须配 cache_clear fixture」。
- **依据**：当前测试用 `auth_module.settings` 替换 patch 是迂回做法，可读性较差；正路是 monkeypatch `Settings.X` 字段 + `cache_clear()`。

### N3. LocalIdP unknown user 错误信息回显 username，可被用户枚举

- **位置**：`backend/app/services/auth.py:101`
- **当前**：
  ```python
  raise AuthError(f"Unknown user: {username}")
  ```
  `Wrong password`（line 104）和 `Unknown user: {x}`（line 101）走两个分支，攻击者通过响应能区分"用户存在 / 不存在"，配合字典攻击枚举所有员工 username。
- **风险**：LocalIdP 受 production 防呆保护，dev 环境只有员工接触，**风险等级低**。Codex 在自审 §B3 / §E1 已经识别，主动提出未来可改"统一返回 Invalid credentials"。
- **建议**（开发期可缓）：
  - 短期：在 `auth.py:101` / `:104` 各加 `# noqa: E501` 风格的注释"intentional info leak — LocalIdP dev only, see N3 in PR-14.md"，留个 trace。
  - 中期：Phase 2 第一个真实 IdP 接入时，把 LocalIdP 这两行合并成 `raise AuthError("Invalid credentials") from None`，同时把 unknown / wrong password 各自的 detail 写到 `logger.debug`（dev 时能看，prod 不暴露）。
- **依据**：常见 OAuth / login endpoint 反枚举最佳实践；OWASP A07:2021 Identification & Authentication Failures。

- **请回答**：
